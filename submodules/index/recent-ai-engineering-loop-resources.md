# Recent AI-First Engineering Loop Resources

Scope: recent material, roughly April-June 2026, from teams and people shaping
AI-first software engineering practice. This file is for deeper mining around
prompt writing, context engineering, harness engineering, workflows, loops, and
agent orchestration.

## Why This Exists

The strongest recent signal from frontier coding-agent builders is that prompt
writing is becoming part of a larger discipline:

- write compact intent and policy;
- give agents dynamic context, tools, and environments;
- put repeated prompting inside loops;
- verify work with tests, reviewers, traces, and budget caps;
- persist lessons as skills, workflow files, memories, or issue-tracker state.

This does not mean prompts no longer matter. It means prompts are no longer the
whole mechanism.

## Boris Cherny And "Writing Loops"

Primary-ish source trail:

- WorkOS hosted the Acquired Unplugged interview with Boris Cherny, Head of
  Claude Code at Anthropic.
  - `https://workos.com/blog/boris-cherny-claude-code-acquired-interview-takeaways`
  - `https://www.youtube.com/watch?v=RkQQ7WEor7w`
- WorkOS' recap says Claude Code went from writing 10-20% of Boris's code to
  replacing his IDE, and that he now writes automated workflows that prompt
  Claude and decide what to build next.
- Widely shared transcript snippets quote the transition:
  - old mode: hand coding with autocomplete;
  - middle mode: running 5-10 Claude sessions in parallel and prompting them;
  - current mode: loops prompt Claude and figure out what to do.

Working interpretation:

A "loop" is not a single prompt. It is a small control system around an agent:

1. discover or receive work;
2. gather current context;
3. prompt an agent or subagent;
4. inspect the result;
5. run checks, tests, review, or another verifier;
6. decide whether the task is done, needs another pass, or needs a human;
7. persist state so the next iteration does not start cold;
8. repeat within explicit limits.

In other words, the human stops being the person typing every follow-up prompt.
The human writes the system that decides which prompt to send next and when to
stop.

Claude Code has concrete primitives that make this less metaphorical:

- **agent loop:** Claude receives a prompt plus context, decides on text or tool
  calls, tools run, results feed back, and the process repeats until no more
  tool calls are needed;
- **`/goal`:** the user writes a completion condition and Claude keeps taking
  turns until a small evaluator model says the condition holds;
- **`/loop`:** the user schedules a prompt to run repeatedly on an interval,
  useful for babysitting PRs, builds, deployments, or queues;
- **dynamic workflows:** Claude writes a workflow script for a large task; the
  script owns branching, subagent fanout, intermediate results, and quality
  patterns such as adversarial review before final synthesis.

Why it matters for a prompt-writing skill:

- The prompt becomes a component inside a loop, not the whole artifact.
- The durable object may be a skill, `WORKFLOW.md`, cron task, issue state,
  checker script, or subagent config.
- The key prompt-writing question changes from "what should I say?" to "what
  should the loop observe, decide, ask, verify, and remember?"
- Prompt concision matters more because the loop will reuse the prompt many
  times; noisy instructions compound cost and failure.
- The loop must have halt conditions: max iterations, no-progress detection,
  token/dollar budget, and human escalation.

Open questions for deeper mining:

- What exactly did Boris mean by "loops" in his own current setup: Claude Code
  `/loop`, dynamic workflows, custom scripts, scheduled tasks, or all of these?
- What are the minimal primitives needed for a good loop: task queue, state
  store, agent runner, verifier, budget guard, scheduler?
- What prompt surfaces inside a loop should remain human-written versus
  generated by the loop?
- What examples exist of bad loops that amplify prompt sludge or run forever?
- What should a prompt-writing skill teach differently if its target artifact is
  a loop rather than a one-shot prompt?

## Patterns To Mine Next

1. **Loop anatomy:** scheduler, task source, context builder, prompt renderer,
   agent runner, verifier, state store, halt policy, escalation path.
2. **Prompt surfaces inside loops:** seed prompt, per-iteration prompt,
   verifier prompt, summarizer prompt, memory update prompt, human handoff
   prompt.
3. **Failure controls:** bounded iteration, no-progress detection, budget caps,
   tool permission policy, sandbox, review gates, rollback/branch isolation.
4. **Skill relationship:** loops should call small, named, tested skills instead
   of embedding long instructions inline.
5. **Prompt-writing implication:** teach agents to design the smallest prompt
   that lets the loop observe and decide correctly, then move all deterministic
   enforcement into tools, schemas, tests, or workflow policy.
